This is actually a significant milestone. We now have **true two-way multiplayer synchronization**. More importantly, you've implemented it in a way that's much closer to how Fusion is intended to be used than the original RPC approach.

Let's go through everything from the beginning, assuming you're 15 years old and have never seen Fusion before.

# First, what is Fusion actually doing?

Imagine two kids are playing chess.

One is at your house.

One is at your friend's house.

Each has their own chess board.

```
YOU                     FRIEND

♜♞♝♛♚♝♞♜              ♜♞♝♛♚♝♞♜
♟♟♟♟♟♟♟♟              ♟♟♟♟♟♟♟♟
........              ........
........              ........
........              ........
........              ........
♙♙♙♙♙♙♙♙              ♙♙♙♙♙♙♙♙
♖♘♗♕♔♗♘♖              ♖♘♗♕♔♗♘♖
```

There is **NOT one chessboard**.

There are **two boards**.

Fusion's job is simply:

> Keep both boards looking identical.

That's networking.

# What is a NetworkObject?

Anything that Fusion synchronizes is called a **NetworkObject**.

In your project,

```
NetworkGameManager
```

is a NetworkObject.

It exists on both machines.

```
Unity Editor

NetworkGameManager
```

↓

Network

↓

```
Windows Build

NetworkGameManager
```

They are copies of the same object.

Not literally the same object.

Two copies.

Fusion keeps them synchronized.

---

# What does Shared Mode mean?

Fusion has different networking models.

Shared Mode is like everyone sharing one notebook.

Instead of having one big server controlling everything,

each player owns certain objects.

For example

White owns

```
White pawn movement
```

Black owns

```
Black pawn movement
```

Fusion then synchronizes those changes.

---

# What is State Authority?

This is probably the most important concept.

Think of it like ownership.

Suppose your notebook belongs to you.

Only YOU are allowed to erase things in it.

That's State Authority.

```
Host

HasStateAuthority = true
```

```
Client

HasStateAuthority = false
```

Only the authority is allowed to modify Networked variables.

---

# Why couldn't Black update LastMove?

Originally your code did

```csharp
LastMove = move;
```

from both players.

But Fusion says

"No."

Only the owner may change Networked properties.

So Black was trying to edit something it didn't own.

Fusion ignored it.

That's why only White's moves worked.

---

# How did we fix that?

Instead of letting Black edit LastMove,

Black now says

> "Hey owner, please write this for me."

That is exactly what the RPC does.

```
Black

↓

RPC

↓

Host

↓

Host updates LastMove
```

Only the owner modifies the Networked variable.

That follows Fusion's rules.

---

# What is a Networked property?

This

```csharp
[Networked]
public NetworkMove LastMove { get; set; }
```

means

Fusion automatically synchronizes this value.

Imagine it like Google Docs.

You type

```
Hello
```

Your friend's screen updates automatically.

You don't write networking code.

Google Docs sends it.

Fusion does the same thing.

When

```
LastMove
```

changes,

Fusion sends it to everyone.

Fusion sends one suitcase instead of many individual items.

# Why did we add Sequence?

Imagine White moves


Then later


Fusion needs to know

"Is this a new move?"


Very common in networking.

# Why did we add Sender?

This solved your biggest bug.

Imagine

Black moves.

Black already moved his own pawn.

A few milliseconds later,

Fusion tells everyone

```
Black moved.
```

Without Sender,

Black thinks

"Oh, I should move that pawn."

But...

the pawn already moved.

So

```
Remote piece not found
```

because the pawn isn't there anymore.

---

Now imagine we include

```
Sender = Black
```

When the message arrives,

Black checks

```
Was I the sender?
```

Yes.

Then Black ignores it.

White checks

```
Was I the sender?
```

No.

White performs the move.

Perfect.

---

# Why didn't HasStateAuthority work?

Originally we had

```csharp
if (HasStateAuthority)
    return;
```

That only skips the Host.

But Black is not the Host.

So Black replayed its own move.

Which caused

```
Remote piece not found
```

because the pawn had already moved.

Changing it to

```csharp
if (LastMove.Sender == Runner.LocalPlayer)
    return;
```

is much smarter.

Now whoever made the move ignores it.

Host or Client.

---

# Why do we call MoveTo() locally first?

Suppose White moves.

White immediately updates its own board.

Otherwise there would be delay.

Imagine pressing a button in a game.

If it waited 200ms before moving,

the game would feel terrible.

Instead

```
Player drags pawn

↓

Local board updates immediately

↓

Fusion tells everyone else
```

Very responsive.

---

# Why does the remote player call ExecuteRemoteMove()?

The remote board never saw the mouse click.

So it has to recreate what happened.

It receives

```
from = 1,6

to = 1,4
```

Then it says

```
Find pawn

Move pawn

Update board
```

That's exactly what ExecuteRemoteMove() does.

---

# What is an RPC?

RPC means

**Remote Procedure Call**

Think of it like making a phone call.

Black says

"Host, please do this."

Host receives

```
RPC_SendMoveToAuthority()
```

and executes it.

Only the Host receives it because we declared

```csharp
[Rpc(RpcSources.All, RpcTargets.StateAuthority)]
```

meaning

Anyone can call it.

Only the State Authority executes it.

# If an interviewer asks, "How did you synchronize chess moves using Fusion?"

A good answer would be:

> "Each player executes their own move immediately for responsiveness. I represent the move as a custom `NetworkMove` struct marked as a `[Networked]` property on a replicated `NetworkGameManager`. The host, which has State Authority, is the only instance that writes to this property. If a non-authoritative client makes a move, it sends an RPC to the authority, which updates the networked state. Fusion replicates that state to every client, and an `OnChangedRender` callback applies the move on remote boards. I also included the originating `PlayerRef` in the move data so the client that originally made the move ignores its own replicated update, preventing duplicate execution."

That's a concise explanation that demonstrates you understand both **how** it works and **why** it was designed that way.

It will make the rest of the multiplayer features much easier to build and maintain.
